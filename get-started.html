Get Started

What is Engineer?

Engineer is a powerful command-line tool to help track and consolidate SharePoint configuration changes in any number of environments. It’s like version control for site columns, content types, lists, views, and more.

Engineer is inspired by Laravel migrations, and is made possible by PnP-JS-Core, node-pnp-js, and CSOMNode.

Installation

You’ll need Node.js and npm before you begin. Once you’re ready, you can install Engineer using the npm install command.

{% highlight sh %} npm i -g sp-engineer {% endhighlight %}

Once installed, you can type engineer into any console prompt to run Engineer commands. Use engineer -h to see a list of commands.

Start a New Project

Now that Engineer is installed, let’s initialize a new project.

{% highlight sh %} engineer init {% endhighlight %}

The engineer init command creates .gitignore and env.js files in the current working directory. env.js contains the SharePoint site URL as well as authentication information for your target SharePoint environment. Any authentication configuration supported by node-sp-auth can be used as the auth settings in your env.js file.

Install Engineer Lists

Once your env.js file is set up, you’re ready to install Engineer lists to your target SharePoint environment.

{% highlight sh %} engineer install {% endhighlight %}

That’s it. You’re ready to start making configuration changes to SharePoint.

Migrations

Engineer uses migrations to track configuration changes made to SharePoint. You can use migrations to create a queue of tasks that are executed in order on any number of target environments. Think of migrations like source control for your configuration operations.

New Migration

{% highlight sh %} engineer make my-first-migration {% endhighlight %}

This command will create a migrations folder in the current working directory, and a new file called YYYYMMDDHHMMSS-my-first-migration.js in that directory (YYYYMMDDHHMMSS is replaced by the current UTC timestamp). Feel free to open this file to see what’s inside. By default, new migrations are configured to create a new list called My List when migrated.

Migration Status

You can check the current migration status at any time using the status command.

{% highlight sh %} engineer status ┌───────────────────────────────────┬─────────┐ │ 20171025135627-my-first-migration │ Pending │ └───────────────────────────────────┴─────────┘ {% endhighlight %}

You can see that our new migration is currently in Pending status. Use the migrate command to activate our new migration.

Migrate

Use the migrate command to execute all pending migrations.

{% highlight sh %} engineer migrate {% endhighlight %}

Once migrated, you’ll see a new list called My List has been created on your SharePoint site. This migration is now in the Migrated status, and will not be activated again during future migrations.

Roll Back

Use the rollback command to roll back all active migrations.

{% highlight sh %} engineer rollback {% endhighlight %}

Once rolled back, My List is deleted from the SharePoint site and the migration is set to Pending status once again.

Working with Multiple Environments

You can create copies of env.js, allowing you to store authentication and configuration for multiple SharePoint environments.

{% highlight sh %} cp env.js env-dev.js {% endhighlight %}

Once copied, open up the new env-dev.js file and make any needed changes for that environment. Use Engineer’s config option to specify which environment file you are using when executing commands.

{% highlight sh %} engineer -c env-dev.js status {% endhighlight %}

This command will check the current migration status using the authentication and connection information found in env-dev.js.
